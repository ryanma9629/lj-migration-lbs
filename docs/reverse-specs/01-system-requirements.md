# 旧系统需求说明书

## 1. 文档定位

本文档是一份面向迁移重建的、从现有实现反向推导出来的系统需求说明书，用于指导将当前 legacy JSP + Wabacus 应用迁移为 Spring Boot + Vue 等现代前后端分离架构。

它不是原始立项阶段形成的上游需求文档。
它是一份基于以下证据反推得到的需求文档：

- 当前源代码
- 当前 DDL 与表结构
- 已恢复的运行行为
- 为打通主链路而补充的 mock 数据
- 已形成的 reverse-spec 模块说明

本文档的目标，是在允许技术实现彻底替换的前提下，尽可能保留原系统的业务语义、流程规则、角色边界、数据含义与迁移约束。

## 2. 置信度标记

- `Confirmed`
  - 可由现有源码、SQL、运行结果或已验证 demo 数据直接支持
- `Inferred`
  - 基于命名、重复模式、局部证据做出的高概率推断
- `Open`
  - 以当前代码、表结构和数据尚不能安全还原

## 3. 迁移目标

`Confirmed`

本次迁移不是一个视觉重设计项目。
它的首要目标，是在现代技术栈上重建旧系统，并在当前证据允许的范围内尽可能保留原有业务逻辑。

新系统应当：

- 保留原系统的业务目标与模块边界
- 保留认领、审批、调整、查询、权限等核心业务规则
- 将旧系统中偏 DB2/单体页面驱动的实现迁移为现代服务化架构
- 支持后续用真实历史数据或生产近似数据替换当前 mock 数据

新系统不需要保留：

- JSP 页面结构
- Wabacus 的配置方式
- 旧 Java 回调类的组织形式
- 旧菜单渲染机制本身这种实现细节

## 4. 业务问题定义

`Confirmed`

该系统是一套银行业绩归属与审批平台。
它的核心目标，是判定某一笔业务记录最终应当归属于哪个人、哪个网点、哪个机构，以及按什么比例归属。

用更直白的业务语言表达，这套系统解决的问题是：

> 一笔业务业绩，到底应该算到谁头上，按多少比例计算，并经过谁审批后生效？

系统面向多个业务条线。
各业务条线都会产生待认领的业务记录，用户对这些记录进行认领、分配归属比例、提交审批，审批通过后再进入后续查询、调整与考核相关使用阶段。

## 5. 业务范围

### 5.1 当前明确在范围内的能力

`Confirmed`

从现有代码可以明确看出，系统至少包含以下能力域：

1. 业务记录认领发起
2. 归属比例录入与暂存
3. 认领提交进入审批
4. 单级或多级审批与驳回
5. 已认领结果的后续调整
6. 个人认领与调整结果查询
7. 用户、组、权限、菜单等系统管理

### 5.2 当前代码中出现的业务条线

`Confirmed`

当前 legacy 代码中可以识别出多个业务条线模块，包括但不限于：

- 存款
- 贵金属
- 基金
- 理财/金融交易
- 第三方存管
- 代发工资
- 保险
- 贷款
- 公积金相关业务

`Inferred`

不同业务条线在总体操作模型上高度相似，但在以下方面存在差异：

- 来源表不同
- 查询字段不同
- 审批范围不同
- 业务校验细节不同

## 6. 主要角色

### 6.1 认领人

`Confirmed`

职责：

- 查找可认领的业务记录
- 填写归属比例和备注
- 暂存认领结果
- 提交进入审批流程

典型约束：

- 只能操作当前个人或组织范围内可见的记录
- 归属比例不能超出允许总额
- 进入后续状态的记录不能被任意反复修改

### 6.2 审批人

`Confirmed`

职责：

- 查看待审批认领记录
- 对认领进行通过或驳回
- 记录审批人身份和审批时间

`Inferred`

不同模块中，审批人可能分为多个层级，例如：

- 初审
- 支行/机构级审批
- 终审

### 6.3 调整发起人

`Confirmed`

职责：

- 对已经认领或已经审批通过的归属结果发起调整
- 将调整结果提交进入新的审批链路

### 6.4 系统管理员

`Confirmed`

职责：

- 维护用户、组、权限、菜单与页面访问关系
- 为业务人员开放不同模块的可见与可操作范围

### 6.5 外部身份提供方

`Confirmed`

代码中存在基于 token 的外部单点登录校验路径。
这说明“登录身份”可以来自系统外部，但“授权、菜单、审批流转”仍主要由本系统内部负责。

## 7. 系统上下文

`Confirmed`

这套系统位于“上游业务数据”与“下游业绩使用/考核使用”之间。

其高层上下文可概括为：

1. 业务记录由外部业务系统产生，或被加载进入业务来源表
2. 本系统将这些记录呈现为可认领对象
3. 用户在本系统内部完成认领、审批、调整
4. 审批通过后的归属结果被后续查询、调整以及可能的考核模块使用

`Open`

当前材料尚不能完整恢复：

- 上游业务数据的完整入库/同步机制
- 下游考核、报表或其他消费方的完整使用链路

## 8. 端到端业务主链路

`Confirmed`

跨多数模块都能看到如下主业务链路：

1. 某笔业务事件或业务凭证进入待认领池
2. 认领人打开该业务记录详情
3. 创建或编辑一个或多个归属明细
4. 填写归属比例与备注并保存
5. 提交认领进入审批
6. 一位或多位审批人进行通过或驳回
7. 审批通过后，该记录进入正式有效的归属结果集合
8. 已生效归属结果后续还可以发起调整
9. 用户通过个人页或管理页查询当前与历史结果

## 9. 功能需求

### 9.1 认领发起

`Confirmed`

系统应允许用户进入某个业务模块，查询或浏览待认领的业务记录。

系统应展示业务条线相关的关键识别信息，例如：

- 凭证号
- 交易号
- 客户信息
- 机构信息
- 产品信息
- 交易日期
- 金额或业务值

`Inferred`

在新系统中，虽然不同业务条线可能仍然使用不同物理表，但在业务建模上应抽象为统一的“可认领业务记录”概念。

### 9.2 归属编辑

`Confirmed`

系统应允许一笔业务记录关联一个或多个归属明细。

系统至少应支持编辑以下内容：

- 认领员工号
- 认领员工姓名
- 认领所属机构
- 归属比例
- 备注

系统应支持在提交前进行暂存。

### 9.3 提交

`Confirmed`

系统应允许用户将已暂存的认领结果提交进入审批流程。

提交动作应当：

- 将认领状态从草稿/暂存类状态推进到待审批状态
- 记录提交时间
- 在原实现有配置时保留或追加提交备注

### 9.4 审批

`Confirmed`

系统应提供审批页面，列出当前审批人权限范围内的待审批记录。

系统应支持：

- 审批通过
- 审批驳回
- 记录审批人身份
- 记录审批时间
- 在原实现有此能力时记录审批意见/驳回原因

`Inferred`

有些模块是一段式审批，直接审批到最终通过。
有些模块至少包含两段审批，例如初审后再进入终审。

### 9.5 调整

`Confirmed`

系统应允许已认领或已审批通过的归属结果进入调整流程。

调整流程应：

- 记录新的比例或新的归属方案
- 保留调整日志或调整明细
- 将调整结果送入新的审批步骤
- 仅在调整审批通过后才真正生效

### 9.6 查询与个人工作台

`Confirmed`

系统应提供查询页，让用户查看：

- 自己提交过的认领记录
- 当前认领状态
- 已通过的归属结果
- 调整申请及其处理结果

`Inferred`

“我的认领”类页面在业务上承担个人归属活动审计轨迹的作用。

### 9.7 系统管理

`Confirmed`

系统应提供以下系统管理能力：

- 用户维护
- 组维护
- 菜单/页面授权
- 组与权限关系维护

现有授权模型是一个以“页面权限 + 分组授权”为核心的模型。

## 10. 通用流程规则

### 10.1 暂存、提交、审批

`Confirmed`

多数认领模块都体现出如下生命周期：

1. 未认领或初始状态
2. 暂存/草稿状态
3. 已提交待审批状态
4. 审批通过状态
5. 审批驳回状态

### 10.2 多级审批

`Confirmed`

部分模块存在如 `21` 这样的中间审批状态，然后才进入最终 `2`。

`Inferred`

这通常意味着存在“初审/支行审批/中间审批层”再到最终审批的结构。

### 10.3 调整流程

`Confirmed`

调整流程会使用独立字段表达调整状态与调整后的值，例如：

- `RVS_STATUS_ID`
- `RVS_RATIO`
- `NEW_CLAIM_STATUS_ID`

调整结果在“保存”时不会立即生效。
它需要经过调整审批通过后才生效。

## 11. 核心业务规则

### 11.1 比例控制

`Confirmed`

归属采用比例分配模式。
系统会反复校验：某笔业务记录已分配比例不能超过可用总额。

`Confirmed`

从代码可见，校验时会把某些“已提交待审批”或“已审批通过”的归属比例一并纳入计算。

### 11.2 重复/冲突认领控制

`Confirmed`

系统会检查某认领人或其他认领人，是否已经在同一笔业务记录上占用了处于有效状态的归属比例。

系统会阻止同一业务事实出现互相冲突的重复认领。

### 11.3 基于状态的可编辑性

`Confirmed`

记录是否允许继续编辑、提交、审批、调整，取决于其当前状态。
处于后续流程状态的记录，不能被自由修改。

### 11.4 基于范围的数据可见性

`Confirmed`

审批页和查询页的数据可见范围至少受以下因素影响：

- 当前用户
- 当前机构
- 机构层级
- 页面/分组权限

### 11.5 审计字段留痕

`Confirmed`

系统会在关键状态流转时记录操作者与时间信息，包括但不限于：

- 提交人
- 审批人
- 初审人
- 终审人

## 12. 状态模型

### 12.1 认领状态

`Confirmed`

当前多模块重复出现的认领状态包括：

- `0`
  - 草稿 / 暂存
- `1`
  - 待审批
- `2`
  - 已审批通过
- `3`
  - 审批未通过 / 驳回
- `5`
  - 某些场景下自动审批通过
- `6`
  - 某些事中场景下自动审批通过
- `21`
  - 初审通过 / 中间审批通过
- `31`
  - 初审未通过 / 中间审批驳回

`Inferred`

不同模块对状态文本的展示可能略有差异，但其底层语义模式足够稳定，可以在新系统中抽象为一类通用认领状态机。

### 12.2 调整状态

`Confirmed`

当前多模块重复出现的调整状态包括：

- `0`
  - 调整草稿 / 未提交
- `1`
  - 调整待审批
- `2`
  - 调整已通过
- `3`
  - 调整未通过
- `21`
  - 多级审批模块中的调整初审通过

## 13. 权限与数据范围需求

`Confirmed`

系统应至少按以下维度控制访问：

- 用户
- 用户组
- 页面权限
- 菜单挂载关系

`Confirmed`

一个用户要想真正“可用”某个业务页面，通常同时需要：

- 菜单/页面可见权限
- 对应业务数据范围可见权限

`Inferred`

在新系统中，即便采用统一 RBAC 模型，也应将“菜单权限”与“业务数据范围”作为两个不同层次来建模，而不应简单混为一体。

## 14. 核心领域模型

### 14.1 业务来源记录

`Confirmed`

业务来源记录代表原始业务事件、交易、凭证、合同或其他待归属业务对象。

### 14.2 认领记录

`Confirmed`

认领记录代表对某笔业务来源记录进行的一次归属操作或归属明细。
它包含认领人、归属比例、流程状态、备注和审计字段等信息。

### 14.3 审批信息

`Confirmed`

当前系统中的审批信息，主要通过以下方式表达：

- 认领记录上的状态字段
- 审批人和审批时间字段
- 部分相关日志/变更表

### 14.4 调整记录

`Confirmed`

调整记录代表对已生效归属结果提出的一次变更申请。
它包含变更后的值、调整状态和审核信息。

### 14.5 安全与组织元数据

`Confirmed`

系统依赖以下主数据/权限数据：

- 用户
- 员工主数据
- 机构主数据
- 用户组
- 页面权限
- 菜单与页面关系数据

## 15. 模块边界需求

`Confirmed`

新系统至少应保留以下类型的模块边界：

1. 按业务条线划分的认领录入模块
2. 按审批阶段或业务条线划分的审批模块
3. 调整模块
4. 个人查询模块
5. 系统管理模块

`Inferred`

虽然新架构可以在内部合并重复的 UI 组件、服务逻辑和数据访问模式，但不应抹平那些对运营方式、权限模型和业务语义有影响的模块差异。

## 16. 外部集成需求

### 16.1 身份认证 / 单点登录

`Confirmed`

代码中存在外部 token 校验路径，说明系统具备接入外部单点登录的能力。
因此，新系统应保留与外部认证系统集成的可能性。

### 16.2 流程引擎

`Confirmed`

当前代码中没有发现明确的第三方 BPM/流程引擎集成痕迹。
认领与审批流转主要是通过页面动作、Java 回调、SQL 更新和状态字段在系统内部实现的。

迁移含义：

- 在没有新的证据之前，新系统应将流程逻辑视为内部业务逻辑实现，而不是外部 BPM 平台依赖

### 16.3 上下游业务数据集成

`Open`

当前代码和现有 DDL 尚不足以完整定义：

- 上游数据如何进入本系统
- 下游哪些系统消费本系统结果

因此，新系统在设计上应保留明确、可配置的集成边界。

## 17. 迁移约束

### 17.1 必须保留的内容

`Confirmed`

迁移时必须尽量保留：

- 认领、审批、调整的业务含义
- 状态流转语义
- 比例分摊规则
- 角色与范围控制行为
- 对“谁在何时做了什么”的可审计性

### 17.2 可以替换的内容

`Confirmed`

迁移时可以替换：

- 前端框架
- 后端框架
- 服务拆分方式
- 持久化实现细节
- SQL 方言与物理表设计
- 菜单渲染实现

### 17.3 当前证据缺口

`Confirmed`

当前逆向需求工作仍受以下限制：

- 缺少完整生产数据
- 缺少完整原始菜单种子数据
- 当前 MySQL 重建环境与源码之间仍存在部分缺表或 schema 漂移
- 某些链路的演示依赖 mock 数据

在证据不完整的情况下，新系统应：

- 保留已确认规则
- 将推断性规则与已确认规则分层管理
- 避免过度拟合当前临时 mock 数据形态

## 18. 当前高置信度样板模块

`Confirmed`

当前最适合作为迁移锚点样板的模块有：

- `claim_gold`
  - 认领 + 审批链路较完整
- `claim_verify_acc`
  - 多级审批特征较典型
- `claimed_change`
  - 调整链路较典型
- `my_claimed`
  - 个人查询链路较典型

这些模块应作为新系统第一轮领域建模和接口设计的优先参考样本。

## 19. 尚未完全还原的问题

`Open`

以下内容目前仍无法完全确定：

- 原生产系统完整菜单树与顶层业务分组
- 原生产环境完整角色体系
- 上游业务数据完整入库机制
- 下游考核、报表或其他消费方的完整链路
- 某些边缘状态码是否在生产中真实使用，还是仅在代码中预留
- 各业务认领模块之间除共性以外的精确业务差异

## 20. 建议的后续逆向文档输出

`Confirmed`

为了支撑后续 Spring Boot + Vue 重建，下一批最有价值的文档是：

1. 各主要业务模块的模块需求分册
2. 跨模块通用状态机/流程字典
3. 角色与权限矩阵
4. 旧表到新领域模型的映射草案
5. mock 数据与真实所需数据之间的缺口报告
